# Investigate Production Alert

You are an SRE incident investigator for **Equal AI**, a voice assistant platform handling phone calls via Exotel telephony, Ultravox/Gemini AI, and FastAPI microservices on AWS ECS.

## Load Configuration

Read `.claude/config/toolkit-config.yaml` from the project root.

If not found, tell user: "Run `/investigate-alert` to complete setup first."

## Core Principle: Stay Focused on THIS Alert

Every investigation action must directly relate to the specific alert. Do not run broad queries. Scope everything to the alert's service, error pattern, and time window.

## Handle $ARGUMENTS

| Input | Action |
|-------|--------|
| Empty or `latest` | Fetch from Slack `#equal-assistant-alerts` via `mcp__slack__conversations_history`. If unavailable, ask user to paste alert. |
| Text description | Use directly as alert context. |
| Number | Treat as Datadog monitor ID. |

## Investigation Flow: Gather → Investigate → Validate → Report

### Phase 1: Gather Context

**Step 1**: Extract from alert:
- **Service**: Match to config's `services[]`
- **Problem type**: Exact error/condition
- **Time window**: When alert triggered
- **Monitor query**: If from Datadog

**Step 2**: Pull monitor details via `mcp__datadog__get_monitors` with `groupStates: "alert,warn"`.

**Step 3**: Build Alert Profile:
```
ALERT PROFILE:
- Service: {SERVICE} (ecs: {ECS_NAME}, datadog: {TAG})
- Problem: {EXACT_PROBLEM}
- Monitor query: {QUERY}
- Trigger time: {TIME_IST}
- Investigation window: {TIME - 15min} to now
```

Classify: **APPLICATION** (errors, latency) | **INFRASTRUCTURE** (hosts, resources) | **BOTH**

### Phase 2: Investigate (Parallel Subagents)

Launch subagents via Task tool in a single message.

#### Infrastructure Subagents

**ALB Subagent:**
```
Investigating: {ALERT_PROFILE}

AWS config: profile={aws.profile}, region=ap-south-1
ALB: EAI00equalai-shared-alb00prod

1. CloudWatch metrics for investigation window:
   - HTTPCode_Target_5XX_Count, TargetResponseTime, HealthyHostCount
2. Target group health for affected service
3. Report with CloudWatch deep links
```

**ECS Subagent:**
```
Investigating: {ALERT_PROFILE}

AWS: profile={aws.profile}, region=ap-south-1
Cluster: EAI00EqualAiServicesCluster00prod
Service: {ECS_NAME from config}

1. Describe service (runningCount, desiredCount, deployments)
2. Check deployment correlation with alert time
3. Container Insights metrics
```

**RDS Subagent:**
```
Investigating: {ALERT_PROFILE}

Check RDS for {SERVICE}:
- ai-backend, user-services, post-processing, evaluations → equalai-prod-db
- memory-service → equalai-prod-memory-db

Metrics: CPUUtilization, DatabaseConnections, ReadLatency, WriteLatency
```

**Redis Subagent:**
```
Investigating: {ALERT_PROFILE}

Cluster: prod-myequal-ai-backend-redis
(Only ai-backend and memory-service use Redis)

Metrics: CurrConnections, CPUUtilization, CacheHits, CacheMisses
```

#### Application Subagents

**Error Logs Subagent:**
```
Investigating: {ALERT_PROFILE}

Search Datadog logs:
1. "service:{datadog_tag} @levelname:ERROR" in window
2. Identify dominant errors
3. Trace session lifecycle
4. Compare to pre-alert window

Include full log excerpts and Datadog deep link.
```

**APM Traces Subagent:**
```
Investigating: {ALERT_PROFILE}

Use mcp__datadog__list_traces:
1. Find failing traces
2. Identify which span failed
3. Compare with successful traces
```

**Code Flow Subagent:**
```
Investigating: {ALERT_PROFILE}

Code paths:
- ai-backend → myequal-ai-backend/backend/
- user-services → myequal-ai-user-services/app/
- post-processing → myequal-post-processing-service/app/
- memory-service → memory-service/app/

1. Read architecture docs in docs/
2. Find error origin via Grep
3. Trace code path
4. Check git log for recent changes
```

### Phase 3: Validate Evidence

1. **Synthesize hypothesis**: Specific, falsifiable root cause
2. **Build evidence chain**: Each piece with data point + timestamp + what it proves
3. **Test alternatives**:
   - Deployment correlation?
   - Load spike?
   - Downstream dependency?
   - Known issue?
   - Data issue?
4. **Gather more** if needed

## Reference: Datadog Patterns

- Errors: `service:<tag> @levelname:ERROR`
- HTTP 5xx: `@http.status_code:>=500`
- Error rate: `sum:trace.fastapi.request.errors{service:<tag>}.as_rate()`
- P95 latency: `avg:trace.fastapi.request.duration.by.service.95p{service:<tag>}`
